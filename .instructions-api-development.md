# API Endpoint Development Skill

## Core Principles

- **Thin Handlers**: Handlers call services; NO business logic
- **Consistent Format**: All responses use standard envelope
- **Input Validation**: Zod schema validation on every endpoint
- **Error Handling**: Throw typed AppError subclasses; middleware handles conversion
- **Authorization**: Every endpoint checks user's workspace membership
- **Logging**: Structured logging with userId and workspaceId
- **Function Composition**: Higher-order functions for handler creation

---

## Request-Response Flow

```
HTTP Request
    ↓
Express Middleware (logging, auth extraction)
    ↓
Route Handler (invoke appropriate handler function)
    ↓
Handler Function:
    1. Extract & validate with Zod
    2. Extract user context from req.user
    3. Call service function with validated data
    4. Format response with standard envelope
    5. Send JSON response with HTTP status
    ↓
If Error: ErrorHandler Middleware
    1. Convert typed AppError to JSON
    2. Respond with consistent error format
    3. Log error details
```

## Handler Function Factory Pattern

Create handlers using higher-order functions that inject dependencies:

```typescript
// handlers/issue.handlers.ts

/**
 * Creates a handler function to create an issue.
 * Dependencies are injected as arguments.
 * 
 * @param issueService - Injected service with all functions
 * @returns Express route handler function
 */
export const createCreateIssueHandler = (issueService: IssueService) =>
  /**
   * POST /api/projects/{projectId}/issues
   * 
   * Create a new issue in the project.
   * 
   * @param req.params.projectId - Project ID
   * @param req.body.title - Issue title (required)
   * @param req.body.type - Issue type (required): epic|story|task|bug|subtask
   * @param req.user - Auth context from middleware
   * @returns 201 with created issue
   * @throws 400 ValidationError if body invalid
   * @throws 403 Forbidden if user not in workspace
   */
  async (req: AuthRequest, res: Response, next: NextFunction) => {
    try {\n      // 1. Extract from request
      const { projectId } = req.params;
      const validated = req.body as z.infer<typeof CreateIssueSchema>;
      const { user } = req;

      // 2. Call service function
      const issue = await issueService.createIssue(
        {
          projectId,
          ...validated,
        },
        user.workspaceId,
        user.id,
      );

      // 3. Format response
      res.status(201).json({
        success: true,
        data: issue,
        meta: {
          timestamp: new Date().toISOString(),
        },
      });
    } catch (error) {
      next(error);
    }
  };

// Usage in routes
export const setupIssueRoutes = (deps: RouteDependencies) => {
  const router = Router();
  const issueService = createIssueService(deps);

  router.post(
    "/:projectId/issues",
    validateRequest(CreateIssueSchema),
    createCreateIssueHandler(issueService),
  );

  router.patch(
    "/:issueId",
    validateRequest(UpdateIssueSchema),
    createUpdateIssueHandler(issueService),
  );

  return router;
};
```

**Benefits**:
- Each handler is a pure function
- Dependencies explicitly injected
- Easy to test with mock services
- Composable and reusable
- No class instantiation overhead
- Natural with Express middleware chain

## Validation Middleware

All inputs validated with Zod before handler sees them:

```typescript
// middleware/validate.ts

/**
 * Creates a validation middleware using Zod schema.
 * Validates req.body, req.params, req.query
 */
export const validateRequest = (schema: z.ZodSchema) =>
  (req: Request, res: Response, next: NextFunction) => {
    try {
      const validated = schema.parse({
        body: req.body,
        params: req.params,
        query: req.query,
      });
      
      // Replace request data with validated data
      req.body = validated.body || req.body;
      req.params = validated.params || req.params;
      req.query = validated.query || req.query;
      
      next();
    } catch (error) {
      if (error instanceof z.ZodError) {
        next(new ValidationError('Request validation failed', error.issues));
      } else {
        next(error);
      }
    }
  };

// Usage in routes
router.post(
  "/",
  validateRequest(CreateSprintSchema),
  createCreateSprintHandler(sprintService),
);
```

## Pagination Patterns

### Offset-based (Simple)

```typescript
export const createListIssuesHandler = (issueService: IssueService) =>
  async (req: AuthRequest, res: Response, next: NextFunction) => {
    try {
      const offset = parseInt(req.query.offset as string) || 0;
      const limit = Math.min(parseInt(req.query.limit as string) || 20, 100);

      const { issues, total } = await issueService.listIssues(
        req.params.projectId,
        {
          offset,
          limit,
          filter: req.query.filter,
          sort: req.query.sort,
        },
        req.user.workspaceId,
        req.user.id,
      );

      res.status(200).json({
        success: true,
        data: issues,
        meta: {
          total,
          limit,
          offset,
          hasMore: offset + limit < total,
          timestamp: new Date().toISOString(),
        },
      });
    } catch (error) {
      next(error);
    }
  };
```

### Cursor-based (Efficient)

```typescript
/**
 * Cursor format: base64(id:createdAt)
 * Example: "aXN1dWVfMTIzOjIwMjQtMDQtMTg="
 */
export const createListCursorIssuesHandler = (issueService: IssueService) =>
  async (req: AuthRequest, res: Response, next: NextFunction) => {
    try {
      const limit = Math.min(parseInt(req.query.limit as string) || 20, 100);
      const cursor = req.query.cursor as string | undefined;

      const { issues, nextCursor } = await issueService.listIssuesCursor(
        req.params.projectId,
        {
          limit,
          cursor,
          filter: req.query.filter,
          sort: req.query.sort,
        },
        req.user.workspaceId,
        req.user.id,
      );

      res.status(200).json({
        success: true,
        data: issues,
        meta: {
          limit,
          cursor: nextCursor,
          hasMore: !!nextCursor,
          timestamp: new Date().toISOString(),
        },
      });
    } catch (error) {
      next(error);
    }
  };
```

## Filtering & Sorting

### Query Parameter Syntax

**Filtering**:
```
GET /api/issues?status=in_progress&priority=high
GET /api/issues?assignee_id=user_123&created_after=2024-01-01
```

**Sorting**:
```
GET /api/issues?sort=created_at:desc
GET /api/issues?sort=priority:desc,created_at:asc (multiple)
```

### Implementation

```typescript
// schemas/queries.ts

export const IssueFilterSchema = z.object({
  status: z.enum(['backlog', 'todo', 'in_progress', 'done']).optional(),
  priority: z.enum(['low', 'medium', 'high', 'critical']).optional(),
  assignee_id: z.string().optional(),
  created_after: z.coerce.date().optional(),
  created_before: z.coerce.date().optional(),
  search: z.string().optional(),
});

export const SortSchema = z.string().regex(/^[a-z_]+(:(asc|desc))?$/);

// handlers
export const createListIssuesHandler = (issueService: IssueService) =>
  async (req: AuthRequest, res: Response, next: NextFunction) => {
    try {
      const filter = IssueFilterSchema.parse(req.query);
      const sort = SortSchema.parse(req.query.sort || 'created_at:desc');

      const issues = await issueService.listIssues(
        req.params.projectId,
        { filter, sort },
        req.user.workspaceId,
        req.user.id,
      );

      res.status(200).json({
        success: true,
        data: issues,
        meta: { timestamp: new Date().toISOString() },
      });
    } catch (error) {
      next(error);
    }
  };
```

## Error Handling in Handlers

**Never catch errors in handlers**. Let middleware handle them:

```typescript
// ❌ BAD: Catching errors loses context
export const createBadHandler = (service: Service) =>
  async (req: AuthRequest, res: Response) => {
    try {
      const result = await service.doSomething();
      res.json(result);
    } catch (error) {
      res.status(500).json({ error: error.message }); // Lost error type!
    }
  };

// ✅ GOOD: Let middleware handle errors
export const createGoodHandler = (service: Service) =>
  async (req: AuthRequest, res: Response, next: NextFunction) => {
    const result = await service.doSomething();
    res.json(result);
    // If service throws, middleware catches it
  };
```

```typescript
      status: jest.fn().mockReturnThis(),
```

## Handler Development Checklist

- [ ] Handler is created via higher-order function (dependency injection)
- [ ] JSDoc comment with @route, @param, @returns, @throws
- [ ] Request validated with Zod schema
- [ ] User context extracted from req.user
- [ ] Service function called (not Prisma directly)
- [ ] Response uses standard envelope format
- [ ] HTTP status code correct (201 for POST, 200 for GET/PATCH, 204 for DELETE)
- [ ] Errors passed to next() middleware (never caught)
- [ ] Meta object includes timestamp
- [ ] Integration tests cover success and error paths
- [ ] Authorization checked by service (not handler)
- [ ] No sensitive data logged

## Common Handler Patterns

### CRUD Handlers

```typescript
// Create
export const createCreateIssueHandler = (s: IssueService) =>
  async (req: AuthRequest, res: Response, next: NextFunction) => {
```

## Key Principles Summary

✅ **DO**:
- Create handlers via higher-order functions
- Extract, validate, call service, format response
- Use Zod for validation
- Pass errors to middleware
- Inject dependencies explicitly
- Write unit tests with mock services
- Use standard response envelopes
- Log structured data

❌ **DON'T**:
- Put business logic in handlers
- Call Prisma directly
- Catch errors (pass to middleware)
- Return raw database objects
- Mix concerns (validation, auth, formatting)
- Use any types
- Log sensitive data
- Skip error type checking
