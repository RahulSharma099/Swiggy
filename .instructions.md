# PMS Backend Development Instructions

## Core Principles

Every code written for this project MUST follow these non-negotiable principles:

### 1. Type Safety

- Enable TypeScript strict mode (`"strict": true`)
- NO `any` types; use generics or `unknown` instead
- All function return types explicitly declared
- Use Zod for runtime validation on ALL external inputs

### 2. Layered Architecture

**Controllers → Services → Repositories → Database**

- **Handlers**: HTTP request handlers; call services only
- **Services**: Business logic functions; orchestrate repositories + external services
- **Repositories**: Database query functions; all Prisma calls encapsulated
- **Database**: Prisma ORM; never raw SQL queries

NEVER put business logic in handlers.
NEVER call Prisma directly from handlers.
NEVER import services inside handlers; pass via route setup.

### 3. Error Handling

- Always throw typed error classes (ValidationError, NotFoundError, etc.)
- NEVER use bare `throw new Error()`
- All errors caught at middleware level
- Consistent error response format with HTTP status codes

### 4. Authorization & Security

- EVERY query must include workspace context
- Impossible to query cross-workspace data
- Log all permission denials
- Validate all inputs with Zod schemas
- Never log sensitive data (passwords, tokens)

### 5. Dependency Injection

- Services accept dependencies as explicit parameters
- Factory functions wrap dependencies for easy reuse
- No global containers; pass deps explicitly through the call chain
- All dependencies mockable for testing
- Makes services testable and reusable

### 6. Audit & Compliance

- EVERY mutation logged to ActivityLog
- Include actor_id, action_type, changed_fields (delta)
- Complete audit trail recoverable
- Soft deletes for data recovery

## Code Patterns

### Service Factory Pattern

Services are created by factory functions that return an object of pure functions. Dependencies are passed explicitly.

```typescript
// services/issue.service.ts
export const createIssueService = (deps: {
  repositories: {
    issue: ReturnType<typeof createIssueRepository>;
    project: ReturnType<typeof createProjectRepository>;
  };
  eventPublisher: EventPublisher;
  logger: Logger;
  workflowEngine: WorkflowEngine;
}) => ({
  // Pure functions (no this binding)
  async createIssue(
    input: CreateIssueInput,
    workspaceId: string,
    userId: string,
  ) {
    // service logic
  },
  async updateIssue(
    issueId: string,
    input: UpdateIssueInput,
    workspaceId: string,
    userId: string,
  ) {
    // service logic
  },
});

// type inferred automatically
export type IssueService = ReturnType<typeof createIssueService>;
```

**Benefits**:

- No class overhead
- Each function is independently testable
- Easy to compose and combine services
- Dependencies explicit, not hidden in constructor
- Works naturally with async/await

### Old Pattern (Not Used)

Don't do this:

```typescript
export class IssueService {
  constructor(
    private prisma: PrismaClient,
    private logger: Logger,
    private issueRepository: IssueRepository,
    private eventPublisher: EventPublisher,
    private workflowEngine: WorkflowEngine,
    private authService: AuthService,
  ) {}

  async createIssue(
    input: CreateIssueInput,
    workspaceId: string,
    userId: string,
  ): Promise<Issue> {
    // 1. Authorize
    await this.authService.validateWorkspaceAccess(userId, workspaceId);

    // 2. Validate
    this.validateIssueHierarchy(input);

    // 3. Create
    this.logger.info("Creating issue", { title: input.title, workspaceId });
    const issue = await this.issueRepository.create(input);

    // 4. Emit event for real-time
    await this.eventPublisher.emit("issue_created", issue);

    // 5. Log activity
    await this.activityRepository.log({
      issueId: issue.id,
      actionType: "created",
      actor_id: userId,
      changedFields: {},
    });

    return issue;
  }
}
```

### Controller Route Pattern

```typescript
router.post(
  "/:projectId/issues",
  validateRequest(CreateIssueSchema), // Validate with Zod
  async (req: AuthRequest, res: Response, next: NextFunction) => {
    try {
      const { projectId } = req.params;
      const validated = req.body as z.infer<typeof CreateIssueSchema>;

      const issue = await issueService.createIssue(
        {
          projectId,
          ...validated,
        },
        req.user.workspaceId,
        req.user.id,
      );

      res.status(201).json({
        success: true,
        data: issue,
        meta: { timestamp: new Date().toISOString() },
      });
    } catch (error) {
      next(error); // Let error middleware handle
    }
  },
);
```

### Repository Query Pattern

```typescript
class IssueRepository {
  async findByIdInWorkspace(
    issueId: string,
    workspaceId: string,
  ): Promise<Issue | null> {
    return this.prisma.issue.findFirst({
      where: {
        id: issueId,
        project: { workspaceId }, // Workspace isolation
        deletedAt: null,
      },
      include: {
        assignee: { select: { id: true, name: true } },
        comments: true,
      },
    });
  }

  async moveToSprint(
    issueIds: string[],
    sprintId: string,
    workspaceId: string,
    userId: string,
  ): Promise<number> {
    return this.prisma.$transaction(async (tx) => {
      // Verify sprint exists in workspace
      const sprint = await tx.sprint.findFirst({
        where: { id: sprintId, project: { workspaceId } },
      });
      if (!sprint) throw new NotFoundError("Sprint", sprintId);

      // Update issues
      const result = await tx.issue.updateMany({
        where: {
          id: { in: issueIds },
          project: { workspaceId },
        },
        data: { sprintId, version: { increment: 1 } },
      });

      // Log activities
      await Promise.all(
        issueIds.map((id) =>
          tx.activityLog.create({
            data: {
              issueId: id,
              actionType: "moved",
              changedFields: { sprintId },
              actorId: userId,
            },
          }),
        ),
      );

      return result.count;
    });
  }
}
```

## Validation Rules

### ALWAYS use Zod for external inputs

```typescript
// ✅ GOOD
export const CreateIssueSchema = z.object({
  type: z.enum(["epic", "story", "task", "bug", "subtask"]),
  title: z.string().min(1).max(200),
  description: z.string().max(5000).optional(),
  priority: z.enum(["low", "medium", "high", "critical"]).optional(),
  storyPoints: z.number().int().min(1).max(100).optional(),
  parentId: z.string().cuid().optional(),
  customFields: z.record(z.unknown()).optional(),
});

// In controller
router.post("/", validateRequest(CreateIssueSchema), async (req, res) => {
  const validated = req.body; // Guaranteed valid
});
```

### Custom validators for complex rules

```typescript
// Validate issue hierarchy constraints
export const validateIssueHierarchy = z
  .object({
    type: z.enum(["epic", "story", "task", "bug", "subtask"]),
    parentType: z.enum(["epic", "story", "task", "bug", "subtask"]).optional(),
  })
  .refine(
    (data) => {
      // Subtasks can't have subtask parents
      if (data.type === "subtask" && data.parentType === "subtask")
        return false;
      // Only epics can be epic root
      if (data.type !== "epic" && !data.parentType && !data.epicId)
        return false;
      return true;
    },
    { message: "Invalid issue hierarchy" },
  );
```

## Testing Requirements

### All new code MUST have tests

- **Services**: 80%+ line coverage
- **Repositories**: 70%+ (focus on complex queries)
- **API endpoints**: Critical paths (CRUD, transitions)
- **Utilities**: 90%+ (error handling, validators)

### Test Template

```typescript
describe("IssueService", () => {
  let issueService: IssueService;
  let mockRepository: jest.Mocked<IssueRepository>;

  beforeEach(() => {
    mockRepository = {
      findById: jest.fn(),
      create: jest.fn(),
      update: jest.fn(),
      delete: jest.fn(),
    };
    issueService = new IssueService(mockRepository);
  });

  describe("createIssue", () => {
    it("should create issue and return it", async () => {
      // Arrange
      const input = { type: "story", title: "Test" };
      const expected = { id: "123", ...input };
      mockRepository.create.mockResolvedValue(expected);

      // Act
      const result = await issueService.createIssue(input);

      // Assert
      expect(result).toEqual(expected);
      expect(mockRepository.create).toHaveBeenCalledWith(input);
    });

    it("should throw ValidationError on invalid input", async () => {
      await expect(
        issueService.createIssue({ type: "invalid" } as any),
      ).rejects.toThrow(ValidationError);
    });
  });
});
```

## Database Query Best Practices

### ❌ ANTI-PATTERNS

```typescript
// N+1 queries
const issues = await prisma.issue.findMany({ where: { projectId } });
for (const issue of issues) {
  issue.assignee = await prisma.user.findUnique({
    where: { id: issue.assigneeId },
  });
}

// Missing workspace context
const issue = await prisma.issue.findUnique({ where: { id } });

// Not using transactions for multi-step operations
await prisma.issue.update({...});
await prisma.activityLog.create({...});  // Could fail; activity not logged if issue update fails

// Not incrementing version for optimistic locking
await prisma.issue.update({ data: { status: 'done' } });  // Doesn't increment version
```

### ✅ PATTERNS

```typescript
// Include relationships
const issues = await prisma.issue.findMany({
  where: { projectId },
  include: {
    assignee: { select: { id: true, name: true } },
    sprint: true,
  },
});

// Always include workspace context
const issue = await prisma.issue.findFirst({
  where: {
    id,
    project: { workspaceId },  // Prevents cross-workspace access
    deletedAt: null,           // Ignore soft-deleted
  },
});

// Use transactions for consistency
await prisma.$transaction(async (tx) => {
  await tx.issue.update({...});
  await tx.activityLog.create({...});  // Both succeed or both fail
});

// Increment version for optimistic locking
await prisma.issue.updateMany({
  where: { id, version: currentVersion },
  data: { status: 'done', version: { increment: 1 } },
});
```

## Logging Standards

```typescript
// ❌ BAD
console.log("User login:", user);
logger.info("Creating issue", { user, input, database, config });

// ✅ GOOD
logger.info("User login", { userId: user.id, workspace: user.workspaceId });
logger.info("Creating issue", {
  title: input.title,
  projectId: input.projectId,
  userId,
});
logger.error("Database error", { error: err.message, userId });
```

**NEVER log**: passwords, tokens, full user objects, full request bodies

**ALWAYS log**: userId, workspaceId, actionType, status codes, error codes

## API Response Format

All responses MUST follow this format:

```typescript
// Success
{
  "success": true,
  "data": { /* resource */ },
  "meta": {
    "timestamp": "2024-04-18T12:00:00Z"
  }
}

// Error
{
  "success": false,
  "error": "Descriptive message",
  "code": "ERROR_CODE",
  "details": { /* optional */ },
  "meta": {
    "timestamp": "2024-04-18T12:00:00Z"
  }
}

// List with pagination
{
  "success": true,
  "data": [ /* items */ ],
  "meta": {
    "total": 500,
    "limit": 20,
    "offset": 0,
    "hasMore": true,
    "timestamp": "2024-04-18T12:00:00Z"
  }
}
```

## Performance Guardrails

- **Query timeout**: 5 seconds (throw TimeoutError)
- **Response size**: < 5MB uncompressed
- **Batch size**: Max 100 items in updateMany
- **Database connections**: Min 20, Max 50
- **Cache TTL**: 1 hour for project settings, 30 min for user permissions

## Security Checklist

- [ ] Authorization on every query/mutation
- [ ] No workspace-crossing queries
- [ ] Passwords hashed with bcrypt (salt rounds: 12)
- [ ] JWT tokens with 15-min expiry + refresh token (7 days)
- [ ] HTTPS enforced (TLS 1.2+)
- [ ] CORS configured to allowed origins only
- [ ] Rate limiting: 100 req/min per user
- [ ] No SQL injection (Prisma parameterized queries)
- [ ] CSRF token for state-changing operations (if using cookies)
- [ ] No sensitive data in API responses
