# Service Development Guidelines

## Purpose

Services contain all business logic. They orchestrate repositories, validate rules, emit events, and maintain audit trails. Services are composed of pure functions that can be tested independently and combined flexibly.

---

## Core Service Factory Pattern

Services are factory functions that return an object of pure functions. All dependencies are passed explicitly.

```typescript
// services/issue.service.ts

/**
 * Creates an issue service with all dependencies injected.
 *
 * @param deps - Injected dependencies
 * @returns Object containing all issue service functions
 *
 * Responsibilities:
 * - Issue CRUD with validation
 * - Hierarchy validation (epic → story → subtask)
 * - Workflow transition enforcement
 * - Event emission for real-time sync
 * - Activity audit trail
 */
export const createIssueService = (deps: IssueServiceDeps) => {
  return {
    async createIssue(
      input: CreateIssueInput,
      workspaceId: string,
      userId: string,
    ): Promise<Issue> {
      // Implementation
    },

    async updateIssue(
      issueId: string,
      input: UpdateIssueInput,
      workspaceId: string,
      userId: string,
    ): Promise<Issue> {
      // Implementation
    },

    async deleteIssue(
      issueId: string,
      workspaceId: string,
      userId: string,
    ): Promise<void> {
      // Implementation
    },
  };
};

// Type auto-inferred from factory return type
export type IssueService = ReturnType<typeof createIssueService>;

// Dependencies interface
export interface IssueServiceDeps {
  repos: {
    issue: ReturnType<typeof createIssueRepository>;
```

## Function Pattern: Authorization → Validation → Mutation → Event → Audit

Each service function follows this 7-step flow:

```typescript
export const createIssueService = (deps: IssueServiceDeps) => {
  const { repos, eventPublisher, logger, workflowEngine, authService } = deps;

  /**
   * Update issue status with workflow validation.
   *
   * @param issueId - Issue to update
   * @param newStatus - Target status
   * @param expectedVersion - Current version (optimistic locking)
   * @param workspaceId - Workspace context (authorization)
   * @param userId - User ID (audit trail)
   * @returns Updated issue
   * @throws NotFoundError if issue doesn't exist
   * @throws ForbiddenError if user not authorized
   * @throws ConflictError if version mismatch
   * @throws UnprocessableEntityError if invalid transition
   */
  async function updateIssueStatus(
    issueId: string,
    newStatus: string,
    expectedVersion: number,
    workspaceId: string,
    userId: string,
  ): Promise<Issue> {
    // 1️⃣ AUTHORIZE: Can user access this workspace?
    await authService.validateWorkspaceAccess(userId, workspaceId);
    logger.info("Status update initiated", {
      issueId,
      userId,
      newStatus,
    });

    // 2️⃣ FETCH: Get current issue with workspace check
    const currentIssue = await repos.issue.findByIdInWorkspace(
      issueId,
      workspaceId,
    );
    if (!currentIssue) {
      throw new NotFoundError("Issue", issueId);
    }

    // 3️⃣ VALIDATE: Workflow and business rules
    const canTransition = await workflowEngine.canTransition(
      currentIssue.projectId,
      currentIssue.status,
      newStatus,
      currentIssue,
    );
    if (!canTransition) {
      throw new UnprocessableEntityError(
        `Cannot transition from ${currentIssue.status} to ${newStatus}`,
        "INVALID_TRANSITION",
      );
    }

    // 4️⃣ MUTATE: Update with optimistic locking
    const updatedIssue = await repos.issue.updateWithVersionCheck(
      issueId,
      expectedVersion,
      { status: newStatus },
    );
    if (!updatedIssue) {
      throw new ConflictError(
        "Issue was modified by another user",
        "CONFLICT_VERSION_MISMATCH",
      );
    }

    // 5️⃣ SIDE EFFECTS: Execute workflow actions
    await workflowEngine.executeActions(
      currentIssue.projectId,
      currentIssue.status,
      newStatus,
      updatedIssue,
    );

    // 6️⃣ EVENT: Emit for real-time sync
    logger.info("Status updated", { issueId, newStatus });
    await eventPublisher.emit("issue_updated", {
      issueId,
      delta: {
        status: { from: currentIssue.status, to: newStatus },
      },
    });

    // 7️⃣ AUDIT: Log to activity trail
    await repos.activity.log({
      issueId,
      projectId: currentIssue.projectId,
      actionType: "transitioned",
      changedFields: {
        status: { from: currentIssue.status, to: newStatus },
      },
      actorId: userId,
    });

    return updatedIssue;
  }

  // Return all service functions
  return {
    updateIssueStatus,
    // ... other functions
  };
};
```

**Benefits of functional pattern**:

- Each function is pure and independently testable
- Dependencies explicit and visible at call site
- Easy to compose functions together
- Natural with async/await
- No class overhead or binding issues
- Easy to mock in tests

## Creating New Service Functions

Follow this checklist for EVERY function:

### 1. Documentation (JSDoc)

```typescript
/**
 * Moves issues to a sprint.
 *
 * @param issueIds - Issue IDs to move
 * @param sprintId - Target sprint ID
 * @param workspaceId - Workspace context (authorization)
 * @param userId - Actor performing the action
 * @returns Promise<number> - Count of issues moved
 * @throws NotFoundError if sprint doesn't exist
 * @throws UnauthorizedError if user not in workspace
 * @throws ValidationError if issues already in sprint
 *
 * @example
 * const moved = await issueService.moveToSprint(
 *   ['ISSUE-1', 'ISSUE-2'],
 *   'SPRINT-123',
 *   'WS-123',
 *   'USER-456'
 * );
 */
```

### 2. Authorization Check

```typescript
// Every method must verify user's workspace membership
await this.authService.validateWorkspaceAccess(userId, workspaceId);
this.logger.info("Operation initiated", { userId, workspaceId });
```

### 3. Input Validation

```typescript
// Validate custom logic beyond Zod (controller handles Zod)
if (issueIds.length === 0) {
  throw new ValidationError("At least one issue must be selected");
}

if (issueIds.length > 100) {
  throw new ValidationError("Cannot move more than 100 issues at once");
}
```

### 4. Use Repository for DB Access

```typescript
// Service NEVER calls prisma directly
const moved = await this.issueRepository.moveToSprint(
  issueIds,
  sprintId,
  workspaceId,
  userId,
);
```

### 5. Emit Events

```typescript
// Publish events for real-time sync
await this.eventPublisher.emit("issues_moved", {
  issueIds,
  sprintId,
  count: moved,
});
```

### 6. Audit Trail

```typescript
// Log all mutations
await this.activityRepository.log({
  issueId: null, // Batch operation
  projectId,
  actionType: "batch_moved",
  changedFields: { sprintId, issueIds },
  actorId: userId,
});
```

### 7. Return Value

```typescript
// Always return what client needs (not DB response)
return { moved, issues: updatedIssues };
```

## Error Handling Patterns

### Validate Before Mutating

```typescript
// ❌ BAD: Check after update
const oldAssignee = issue.assignee;
await this.prisma.issue.update({...});
if (oldAssignee && needsNotification) { /* notify */ }

// ✅ GOOD: Check before update
const oldAssignee = issue.assignee;
if (newAssigneeId === oldAssignee?.id) {
  throw new ValidationError('Assignee unchanged');
}
await this.prisma.issue.update({...});
```

### Use Typed Errors

```typescript
// ❌ BAD
if (!sprint) throw new Error("Sprint not found");

// ✅ GOOD
if (!sprint) {
  throw new NotFoundError("Sprint", sprintId);
}

if (!canTransition) {
  throw new UnprocessableEntityError(
    `Cannot transition to ${newStatus}`,
    "INVALID_TRANSITION",
  );
}
```

### Atomic Transactions for Multi-Step Operations

```typescript
async moveAndNotify(issueId: string, sprintId: string) {
  return this.prisma.$transaction(async (tx) => {
    // Both succeed or both fail
    const issue = await tx.issue.update({...});
    await tx.notification.create({...});
    return issue;
  });
}
```

## Testing Service Methods

Every service method MUST have corresponding test:

```typescript
describe("IssueService.updateIssueStatus", () => {
  let issueService: IssueService;
  let mockRepository: jest.Mocked<IssueRepository>;
  let mockWorkflowEngine: jest.Mocked<WorkflowEngine>;
  let mockEventPublisher: jest.Mocked<EventPublisher>;

  beforeEach(() => {
    // Mock dependencies
    mockRepository = { updateWithVersionCheck: jest.fn() };
    mockWorkflowEngine = { canTransition: jest.fn() };
    mockEventPublisher = { emit: jest.fn() };

    issueService = new IssueService(
      mockRepository,
      mockWorkflowEngine,
      mockEventPublisher,
    );
  });

  it("should update status and emit event", async () => {
    // Arrange
    const issue = { id: "i1", status: "todo", version: 1 };
    mockRepository.findByIdInWorkspace.mockResolvedValue(issue);
    mockWorkflowEngine.canTransition.mockResolvedValue(true);
    mockRepository.updateWithVersionCheck.mockResolvedValue({
      ...issue,
      status: "in_progress",
      version: 2,
    });

    // Act
    const result = await issueService.updateIssueStatus(
      "i1",
      "in_progress",
      1,
      "ws1",
      "u1",
    );

    // Assert
    expect(result.status).toBe("in_progress");
    expect(mockEventPublisher.emit).toHaveBeenCalledWith(
      "issue_updated",
      expect.any(Object),
    );
  });

  it("should throw UnprocessableEntityError if transition invalid", async () => {
    // Arrange
    mockRepository.findByIdInWorkspace.mockResolvedValue(issue);
    mockWorkflowEngine.canTransition.mockResolvedValue(false);

    // Act & Assert
    await expect(
      issueService.updateIssueStatus("i1", "invalid", 1, "ws1", "u1"),
    ).rejects.toThrow(UnprocessableEntityError);
  });

  it("should throw ConflictError on version mismatch", async () => {
    // Arrange
    mockRepository.findByIdInWorkspace.mockResolvedValue(issue);
    mockWorkflowEngine.canTransition.mockResolvedValue(true);
    mockRepository.updateWithVersionCheck.mockResolvedValue(null);

    // Act & Assert
    await expect(
      issueService.updateIssueStatus("i1", "done", 99, "ws1", "u1"),
    ).rejects.toThrow(ConflictError);
  });
});
```

## Key Responsibilities

Services SHOULD handle:

- ✅ Business logic validation
- ✅ Authorization checks
- ✅ Workflow enforcement
- ✅ Event emission
- ✅ Activity audit logging
- ✅ Error handling and logging
- ✅ Orchestrating multiple repositories

Services SHOULD NOT handle:

- ❌ HTTP request parsing (controller's job)
- ❌ Database queries (repository's job)
- ❌ Response formatting (controller's job)
- ❌ WebSocket broadcasting (event publisher's job)

## Common Service Classes

### 1. IssueService

- Create/read/update/delete issues
- Fetch issue hierarchy
- Validate parentchild relationships
- Publish issue\_\* events

### 2. SprintService

- Create/update sprints
- Move issues to/from sprints
- Calculate velocity
- Sprint completion with carry-over

### 3. WorkflowEngine

- Check if transition is allowed
- Evaluate transition conditions
- Execute transition actions
- Log transition history

### 4. NotificationService

- Create notifications
- Fetch user notifications
- Mark as read/unread

### 5. CommentService

- Create/update comments
- Parse @mentions
- Fetch comment threads

### 6. SearchService

- Full-text search
- Structured filtering
- Pagination

## Performance Patterns

### Batch Operations

```typescript
// Move 100 issues in one transaction
async moveMultipleToSprint(
  issueIds: string[],
  sprintId: string
): Promise<number> {
  if (issueIds.length > 100) {
    throw new ValidationError('Maximum 100 issues per operation');
  }

  return this.issueRepository.moveToSprint(issueIds, sprintId);
}
```

### Avoid Multiple Queries

```typescript
// ❌ BAD: Multiple queries
async getIssueDetails(issueId: string) {
  const issue = await this.issueRepository.findById(issueId);
  const assignee = await this.userRepository.findById(issue.assigneeId);
  const comments = await this.commentRepository.findByIssue(issueId);
  return { issue, assignee, comments };
}

// ✅ GOOD: Single query with includes
async getIssueDetails(issueId: string) {
  return this.issueRepository.findByIdWithRelations(issueId);
}
```

## Checklist for New Service Method

- [ ] JSDoc comment with @param, @returns, @throws
- [ ] Authorization check (`authService.validateWorkspaceAccess`)
- [ ] Validation for custom business rules
- [ ] Fetch current state (if updating)
- [ ] Check preconditions before mutation
- [ ] Mutate via repository (not prisma directly)
- [ ] Emit event (if change affects other users)
- [ ] Log to activity trail
- [ ] Return appropriate value
- [ ] Test coverage > 80%
- [ ] Error types are specific (not generic Error)
